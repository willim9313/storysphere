## 1. 背景與範圍

事件分析頁（`/books/:bookId/events`）v1 已落地。2026-07-22 的 UX 實測發現：
**資料維度豐富，但清單缺乏組織、批次缺乏彈性、事件之間的關聯完全沒被呈現**。
本次針對「呈現既有資料」與「幾個真缺口」重設計。

**涉及畫面／元件**：
- `EventAnalysisPage`（左 260px 清單 + 內容區）
- `EventListItems`（AnalyzedItem / UnanalyzedItem）
- `BatchEepPanel`（批次生成面板）
- `EventAnalysisDetail`（詳情：hero / 前後狀態 / 參與角色 / 因果 / 影響 / 因素後果 / 引言）
- **新增**：`EventOverviewLanding`（事件總覽落地頁，§3.0）、事件脈絡視圖（§3.5）、事件對比（§3.6）

> ⚠️ **2026-07-22 增補（本次修改的重點）**：事件分析頁**缺少總覽落地頁**，進頁只有被動的
> 「選擇一個事件」空狀態，使用者不知道要幹嘛。角色分析頁已用 `CharacterOverviewLanding`
> 解決——請以它為藍本補上事件版總覽（見 §3.0，優先序最高）。清單組織（§3.1）與跨事件
> 事件脈絡（§3.5）應收攏成這個落地頁裡的視圖，而非各自獨立的畫面。

**明確不在範圍**（已定案，勿設計）：
- **深色主題**——雙主題僅 Warm/Ink 皆淺底，維持。
- **響應式/窄螢幕**——固定雙欄，不做 mobile 退化。
- 後端 API schema、EEP 生成演算法、task polling 機制。

## 2. 資料事實（設計的基礎，勿臆造）

以 `sample-payloads/` 的真實回應為準。重點：

**清單** `event-list.json` — 每筆事件已具備但現況未善用的維度：
- `chapter`：事件所屬章（本書 1–7 章，共 62 事件橫跨全書）
- `importance`：`KERNEL`（核心）/ `SATELLITE`（衛星）/ `null`（未分析）
- `narrativeMode`：`present` / `flashback` / `flashforward` / `parallel`
- `status`：`complete` / `partial`

**詳情** `eep-complete.json` — `eep` 內：
- **`priorEventIds` / `subsequentEventIds`**：⚠️ **不是因果關係**。後端定義為「與本事件
  **共享參與者**、且位於**較早／較晚章節**」的事件（由 participant timeline 推導，
  見 `analysis_service.py:814-828`）。一律**不得稱為「因果」**，且**可能為空**
  （實測一個 Ch.1 核心事件兩者皆空）。§3.5 以此為準。
- `participantRoles[]`：`{ entityId, entityName, role, impactDescription }`
- `stateBefore` / `stateAfter`、`structuralRole`、`thematicSignificance`
- `causalFactors[]`、`consequences[]`、`keyQuotes[]`
- 另有 `causality`（rootCause + causalChain + chainSummary）、`impact`（雙欄）、`summary`

> ⚠️ 示範 payload 的人名/章號/文字僅為**結構參考**，設計中的排版請照真實字串長度分布，
> 但實作一律即時從真實圖譜計算、不寫死示範值。

## 3. 設計需求清單

優先序：**3.0 為最高（骨幹）**；3.1、3.5 為高；3.2、3.3 為中；3.4、3.6 為低。

### 3.0 事件總覽落地頁 EventOverviewLanding【最高，新功能，本次重點】⭐
現況進頁 = 被動的「選擇一個事件」空狀態，毫無方向感。**請比照角色分析頁的
`CharacterOverviewLanding` 補上事件版總覽落地頁**，建立「無選取 → 總覽；選取 → 詳情 + 返回總覽」
的雙態骨架（見 `screenshots/ca-overview-*.png` 為藍本，視覺語言請一致）。

角色頁總覽的結構（要複製的模式）：
- 頂部：標題 + 統計 meta（總數 / 已分析 / 未分析）
- **雙視圖切換**：象限地圖（QuadrantView，二維散佈）／排序清單（RankingView）
- 批次生成入口（top-N / 全部）
- 點卡片 → 進詳情；詳情頂部有「返回總覽」

事件版該放什麼（用既有資料，勿臆造欄位）：
- **「章節 × 重要度」故事骨幹圖**：以章節為橫軸鋪開全書事件，核心事件(KERNEL)放大強調——
  這是象限地圖的事件對應物，一眼看出「這本書的關鍵節點分布在哪幾章」。
- 或**核心事件脈絡線**：用 `eep.priorEventIds` / `subsequentEventIds` 把 KERNEL 事件串成主線
  （與 §3.5 同源，落在同一落地頁；注意這是共享參與者的鄰接，**非因果**）。
- 排序視圖：依章節 / 重要度列出。
- 統計：總數 / 已分析 / 未分析 / KERNEL 數。
- 批次入口（承 §3.3）：只生成核心 / 本章 / 全部。
- 空/未分析情境：整本尚未分析時，落地頁仍要成立（引導去批次生成），不可退回空白。

設計要點：這是**入口的第一印象**，要立刻回答「這頁在幹嘛、我可以做什麼」。

### 3.1 左清單的組織【高】
現況 62 事件只按「已分析 / 尚未分析」兩組平鋪，排序讓開頭全是 Ch.1，
第一眼會誤以為章節壞掉。需求：
- **章節分組 / 折疊**（事件已有 `chapter`）——讓使用者依閱讀順序導覽全書事件。
- **篩選 chip**：重要度（核心 K / 衛星 S）、敘事模式（倒敘/預敘/平行）。
- **排序**：章節序、重要度。
- 「已分析 / 未分析」的區分需保留（可與分組正交，例如章內再分兩態，或用狀態點表達）。
- 設計要點：260px 窄欄內，分組標頭 + 篩選 + 每筆的多維度標記（K/S、章、敘事、狀態點）
  如何不擁擠。現況每筆已塞 `[K/S][名稱][章·敘事][●]`，是密度上限參考。

### 3.5 事件脈絡視圖【高，新功能】⭐（原名「跨事件因果流」，已正名）
現況全是單事件 drill-in，事件之間的關聯完全看不到。需求：
- 用 `eep.priorEventIds` / `subsequentEventIds` 呈現**書級「事件脈絡」**：
  同一批參與者串起的跨章節事件序列。
- ⚠️ **嚴禁稱為「因果」**（見 §2 的資料語意）。副標須明示
  「依共享參與者、跨章節的相鄰事件串接（**非因果推斷**）」。
  真正的因果只在單一事件內部的 `causality.causalChain`（文字步驟），跨不了事件。
- 與既有「時間軸」頁互補（時間軸是敘事時序；此處是**人物在事件間的串接**）。
- **必須有空資料降級態**（prior/subsequent 常為空）：
  全書無足夠關聯 → 空狀態；單事件無關聯 → 「此事件目前無跨章節的關聯事件」。
- 兩處呈現：總覽的一個視圖 + 詳情的「上下文位置」tab。

### 3.2 未分析事件的空狀態【中】
現況選未分析事件只有「事件名 + 建立按鈕」，使用者在花一次 LLM 前看不到內容。需求：
- 空狀態先**預覽該事件對應的原文段落 / chunk**，讓人判斷值不值得分析。
- 保留「建立（生成分析）」主動作。

### 3.3 批次生成的彈性與時間感【中】
現況「一鍵生成全部 EEP」是全有全無、無 ETA。需求：
- **子集觸發**：只生成核心事件 / 只生成本章 / 勾選多筆。
- **粗略耗時/進度預估**（標示為估計，避免精確承諾）。
- 保留現有批次進度條 + live 階段 + 完成 toast + 統計（生成/略過/失敗）的表達。

### 3.4 參與角色的語意與圖例【低】
現況角色色彩分三桶，但 `beneficiary`（受益者）被歸到 witness 色，語意錯。需求：
- 依 `participantRoles[].role` 的真實值重訂色彩桶（發起/行動/受影響/見證…）。
- 加**角色圖例**說明各色語意（現況無圖例）。

### 3.6 事件對比【低，新功能】
- 從清單選兩個事件，並排比較 `stateBefore/After` 與 `impact`。
- 呼應知識圖譜頁已有的 pair mode 概念，視覺語言盡量一致。

## 4. 設計約束（全站既有規範）

- 顏色/字體一律 CSS variable（`var(--*)`），token 見 `02-tokens.css` / `03-DESIGN_TOKENS.md`；
  **新 token 需同步對照表**。禁止硬編碼色碼。
- 雙主題 **Warm（預設暖色）+ Ink（黑白墨色）**，皆淺底——**無深色主題**，兩套皆需成立。
- **不做響應式/窄螢幕**；固定雙欄（左 260px + 內容區）。
- 文案 i18n（zh 為主），用 `i18n/analysis.zh-TW.json` 的真實字串排版。
- 任務狀態一律 2 秒輪詢，無 WebSocket；勿假設即時 push。
- 沿用 v1 已建立的視覺語言（serif 標題、K/S 徽章、階梯因果鏈、雙欄影響等），
  本次是**擴充與重組**，非推倒重來；與角色分析頁 `CharacterAnalysisPage` 維持一致。
- 現有 UI 規格 `docs/UI_SPEC.md §3.5`；重設計後回寫。

> 交給 Claude Design 的參考資訊。本文件描述**現況**（功能契約、資料、互動、限制），
> 供你作為重新設計的底。標「🔒 必須保留」＝功能不可少；標「🎯 可重新設計」＝樣式/互動開放。

## 0. 這次要你做什麼（Brief）

**要的產出**：針對這個畫面**發散出幾個不同方向的創意概念版本**（divergent concepts），
不是一版定稿。我們會先看多個方向、挑出對的路子，再回來做後續細化與討論。
每個版本請講清楚它的**核心主張**（用什麼結構解決體感問題）與**取捨**，方向之間要真的不同，
不要只是同一版的配色變體。

**重新設計的動機（唯一目標）**：現況介面在**兩種體感**上仍不合理，要一起解掉——

1. **操作體感**：切分章節、改章節/段落角色、合併/刪除這些控制用起來彆扭——
   核心的「切分」藏在 hover、改角色要一直點循環、兩層角色概念混淆（見 §7 痛點）。
2. **閱讀審核體感**：這頁本質是「邊讀解析後的書、邊校對結構」，但現在的兩欄
   列表＋段落卡不像在讀一本書，校對時很難順著內容讀下去、順手修結構。

換句話說：**讓「讀過去 → 順手確認/修正結構」變成自然、連貫、低摩擦的體驗**，
操作與閱讀不該互相打架。

**判斷成功的方向感**（給你發想時的北極星，非硬規則）：
- 多數情況使用者只想**快速掃過確認**，只有機器切錯才動手——快速確認要非常順。
- 需要修時，工具就在內容旁邊、可發現、可觸控/鍵盤操作。
- 兩層角色（章節層級 vs 段落層級）在視覺上要能一眼分清在改哪一層。

§7 是我從現況推斷的具體痛點，§8 是功能不可退化的邊界——發想時 §8 的 🔒 要守住，
🎯 全開放。

## 1. 這個畫面是什麼、在哪一步

**元件**：`ChapterReviewPage`（路由 `/upload/:bookId/review?taskId=...`）

書籍上傳後，後端會先解析檔案（PDF/DOCX/TXT/EPUB）→ 自動偵測章節邊界 →
**暫停** pipeline（任務狀態 `awaiting_review`），把控制權交給使用者。這個畫面就是
那個「暫停確認關卡」（HITL, human-in-the-loop）：讓使用者在昂貴的下游分析
（embedding / 知識圖譜 / 摘要）開始**之前**，覆核並修正機器切出來的章節與段落結構。

完整任務流程：
```
pending → running → awaiting_review →（使用者在本頁確認）→ running → done
```

送出後不可回頭；放棄則整本書刪除。所以這一頁是**破壞性下游動作前最後一道人工閘門**。

---

## 2. 使用者要完成的工作（Jobs-to-be-done）

一個人剛丟進一本書，機器猜了章節結構，可能猜得不完美。使用者要在這頁做到：

1. **看懂**機器把書切成了哪些章節、每章的角色（正文 / 目錄 / 序 / 跋 / 其他）
2. **修正章節邊界**：該分沒分（把一章從某段落切成兩章）、不該分（把相鄰章併回去）
3. **修正章節角色**：把被誤當正文的目錄/序/跋改成非正文（非正文不進閱讀頁、不進 KG/摘要）
4. **修正章節標題**
5. **修正段落角色**：把某段標成分隔/小節/題詞/前言等非正文段落
6.（輔助）**一鍵請 AI 找頭尾雜訊**：書首書尾常黏著版權頁/目錄/推薦語/跋，
   偵測切不乾淨，可按鈕讓 LLM 找邊界並自動切出獨立非正文章節，再人工覆核
7. **送出**繼續處理，或**放棄**整本刪除

> 心智模型關鍵：使用者是在**校對／確認**機器的成果，不是從零編輯。多數情況他只是
> 掃一眼按送出；只有機器切錯時才需要動手。設計要讓「一切正確 → 快速確認」很順，
> 同時「需要修」時工具就在手邊。

## 3. 現況版面與互動（功能契約）

目前是**兩欄**桌面版面（未特別處理 RWD / 窄螢幕）：

```
┌──────────────┬──────────────────────────────────────────────┐
│ 章節列表 224px │ 段落卡片區（選中章節的內容）                   │
│ (左 aside)    │ (右 main, flex-1, 可捲動)                     │
│              │                                              │
│ [審核標題]    │  [章節標頭列：編號 · 標題輸入 · 角色鈕 · 刪除鈕] │
│ [N 章]        │                                              │
│              │  [段落卡] ← hover 卡上緣浮出「+ 在此新增章節」  │
│ · 第 1 章     │  [段落卡]    每張卡右上有段落角色切換鈕         │
│ · 第 2 章 ◄選中│  [段落卡]                                     │
│ · 目錄 (斜體)  │  ...                                          │
│ · 第 3 章     │                                              │
│  ...          │                                              │
│              │                                              │
│ [底部固定]    │                                              │
│ [邊界輔助辨識] │                                              │
│ [送出審核]    │                                              │
│ [放棄上傳]    │                                              │
└──────────────┴──────────────────────────────────────────────┘
```

### 左欄 — 章節列表（🔒 功能 / 🎯 樣式）
- 每章一列，顯示**角色感知編號**：只有 `body` 章節計入「第 N 章」且從 1 連號；
  非正文章節不佔編號，改顯示角色標籤（目錄／序／跋／其他）。切換角色時**即時重算編號**。🔒
- 章節有自訂標題時，於編號下方以次要色顯示（truncate）。
- 非正文章節目前以：淡底（`--bg-tertiary`）＋ 斜體 ＋ 灰字（`--fg-muted`）＋ 左側 2px 灰 bar 區分。🎯
- 選中章節：`--entity-con-bg` 底 ＋ `--entity-con-fg` 字 ＋ 加粗。🎯
- 點擊 = 切換右欄顯示的章節。🔒

### 右欄標頭列（🔒 功能 / 🎯 樣式）
- 顯示目前章節編號/角色標籤
- **標題輸入框**（可編輯章節標題）🔒
- **章節角色切換鈕**：點擊循環 `body → toc → preface → afterword → other`（正文→目錄→序→跋→其他）🔒
  - 目前用「點擊循環單一按鈕」（因為沒有現成的 per-row `<select>` 前例）——**這個循環互動明確開放重新設計**，例如下拉選單、分段控制、圖示切換皆可。🎯
- **刪除章節鈕**：刪除本章，段落併入相鄰章（第一章往後併，其餘往前併）；只剩 1 章時不可刪。🔒

### 右欄段落卡片（🔒 功能 / 🎯 樣式）
- 每個段落一張卡，顯示段落文字。
- 若段落含章節標題（`titleSpan`），標題部分以強調樣式前置顯示，其餘為內文。🔒
- **段落角色切換鈕**（卡右上）：body 時顯示「▸」，非 body 時顯示角色標籤（分隔/小節/題詞/前言）。
  點擊在 body ↔ 各非正文角色間循環。🔒（互動樣式 🎯）
- 非正文段落整張卡降透明度（opacity 0.6）＋角色鈕高亮。🎯
- **章節切分**：body 段落卡的**上緣**有一條 hover 才浮現的「+ 在此新增章節」細線按鈕，
  點擊 = 從這個段落把當前章節切成兩章（此段以後成為新的一章）。🔒（這是核心切分互動，
  目前藏在 hover、可發現性低，**重點重新設計對象**）🎯
- 選中的是非正文章節時，整個右欄 main 區給淡底提示。🎯

### 底部動作（🔒）
- **邊界輔助辨識**（AI 輔助）：見 §4。按下後 loading，完成後顯示結果註記，且變為 disabled（一本書只跑一次）。
- **送出審核**：把目前章節結構送後端，導回上傳頁。
- **放棄上傳**：刪除整本書並清 sessionStorage 任務紀錄，導回上傳頁。

## 4. 「邊界輔助辨識」AI 輔助流程（🔒 功能）

專門解決一個難題：**書首/書尾的非正文（版權頁、目錄、序、作者簡介、推薦語、跋、書目）
常常「融進」第一章或最後一章的頭尾**，章節偵測切不出來（因為沒有明顯標題行）。

- 使用者按鈕 → 後端只掃 **body 章節**，從書首、書尾**逐段往內回推**，每段送 LLM 判
  「正文 / 非正文」，讀到第一段真正的故事正文就停（中段不送、省 token）。
- 回傳前附結束界與後附起始界（book-global 段落索引）＋ LLM 判定的角色。
- 前端據此把受影響的 body 章節**切開**，前/後附段落成為獨立的非正文章節，
  **左側章節列表即時更新**（新章節以非正文樣式呈現），供使用者覆核。
- 這只是**建議**，不改文件、不 resume；使用者仍須按「送出審核」才生效。
- 語言無關（不靠關鍵詞表）。無 LLM provider 時回 503，按鈕流程要能顯示「AI 不可用」。

設計上這是「一鍵幫我清乾淨頭尾」的加速器，跟手動切分並存。

## 5. 資料契約（設計可用的欄位 / 必須送回的欄位）

### 載入（GET review-data → 畫面初始狀態）
```ts
chapters: Array<{
  chapterIdx: number;                 // 章節在列表中的序（0-based）
  title: string | null;               // 章節標題（可能沒有）
  role: 'body'|'toc'|'preface'|'afterword'|'other';   // 章節層級角色
  paragraphs: Array<{
    paragraphIndex: number;           // book-global 全域段落索引（跨章唯一、遞增）
    text: string;                     // 段落全文
    role: 'body'|'separator'|'section'|'epigraph'|'preamble';  // 段落層級角色
    titleSpan: [number, number] | null;   // 標題在 text 內的字元起訖，null=此段非標題
    sentences: string[];              // 已斷句（目前 UI 未用，但資料有，可供新設計利用）
  }>;
}>
```

### 送出（POST review ← 使用者確認後）
```ts
{
  chapters: Array<{
    title: string;                    // 章節標題（空字串代表無）
    role: string;                     // 章節層級角色
    startParagraphIndex: number;      // 該章第一個段落的 book-global 索引 = 章節邊界
  }>;
  roleOverrides: Record<string, string>;   // "全域段落索引" → 段落角色；只放被使用者改過的
}
```

> **邊界的本質**：章節邊界 = 每章「起始段落的全域索引」。切分/合併/AI 切邊界，
> 最後都化約成「這串段落要在哪些 `startParagraphIndex` 斷開」。段落角色改動則走
> `roleOverrides`（稀疏，只記被改的）。新設計無論長怎樣，最終要能產出這兩樣。

**角色詞彙表（送 Claude Design 用，需雙語 i18n：zh-TW 主 / en）**
- 章節角色：`body`=正文、`toc`=目錄、`preface`=序、`afterword`=跋、`other`=其他
- 段落角色：`body`=正文（標籤留空）、`separator`=分隔、`section`=小節、`epigraph`=題詞、`preamble`=前言

## 6. 技術與樣式限制（新設計必須落在這些軌道內）

- **技術棧**：React + TypeScript + Tailwind CSS。逐元件 inline style 目前大量使用 CSS 變數。
- **主題 token 制**：所有顏色/字體**一律用 CSS 變數 `var(--*)`**，禁止硬編碼色碼。
  支援亮/暗主題切換（同一組 token 兩套值）。新設計請以語義 token 表達，不要指定具體色值。
  審核頁目前用到的 token：
  `--bg-primary/secondary/tertiary`、`--fg-primary/secondary/muted`、`--border`、
  `--accent`、`--accent-bg`、`--entity-con-bg/fg/border`、`--color-error`。
  （新增任何 token 需同步登記於 `docs/DESIGN_TOKENS.md`。）
- **i18n**：zh-TW 為主、en 需齊備。所有文案走 i18n key（`upload.json` 的 `review.*`），
  不寫死字串。中文字串會比英文長/短，版面要能容納兩者。
- **圖示**：目前用 `lucide-react`（Loader2 / ScanLine / Trash2）。
- **RWD**：目前是桌面兩欄、未處理窄螢幕——**是否要支援平板/手機由新設計決定**（現況＝未支援）。
- **無障礙**：hover-only 的切分按鈕目前對鍵盤/觸控不友善，是已知弱點，新設計宜改善。

## 7. 現況痛點（重新設計想解決的候選問題）

> 以下是我從程式碼與互動推斷的弱點，**非使用者親述的目標**——實際重設計目標請以你的說明為準。

1. **切分動作藏在 hover**：段落卡上緣的「+ 在此新增章節」只有滑鼠移上去才出現，
   可發現性低、觸控/鍵盤不可用——這是本頁最核心的操作卻最不顯眼。
2. **兩種角色概念容易混淆**：章節層級角色（正文/目錄/序/跋/其他）與段落層級角色
   （分隔/小節/題詞/前言）用相似的「循環按鈕」樣式，使用者難分辨在改哪一層。
3. **「循環點擊」改角色不直觀**：要點好幾下才循環到目標角色，且看不到有哪些選項。
4. **看不到全局結構**：左欄只有列表，缺乏「整本書切成幾段、正文 vs 非正文佔比、
   哪裡可能切錯」的概觀；長書要一章章點進去才知道。
5. **切分/合併不對稱**：切分在右欄段落層級操作，合併（刪除章節併入相鄰）在標頭列，
   兩個相關動作分散。
6. **確認 vs 編輯無層次**：多數使用者只想快速確認，但畫面把所有編輯控制都平鋪出來。

## 8. 給設計的明確邊界

**🔒 一定要能做到（功能不可退化）**
- 看到章節列表與角色感知編號、切換檢視章節
- 改章節標題、改章節角色（5 種）、刪章節（含段落併入相鄰）
- 在任一 body 段落處把章節切成兩段
- 改段落角色（5 種）
- 觸發「邊界輔助辨識」AI 並覆核其結果、處理 AI 不可用(503)
- 送出審核 / 放棄上傳（破壞性，需明確）
- 產出 §5 的送出資料（章節 startParagraphIndex 陣列 + 段落 roleOverrides）

**🎯 完全開放重新設計**
- 版面結構（不必維持兩欄）、RWD 策略
- 章節/段落角色的選取控制（循環鈕 → 下拉/分段控制/選單皆可）
- 切分互動的呈現與可發現性
- 全局結構概觀、確認 vs 編輯的資訊層次
- 視覺語言（在 token 制與亮/暗主題前提下）

**其他約束**：CSS 變數主題制、雙語 i18n、React+Tailwind、lucide 圖示。

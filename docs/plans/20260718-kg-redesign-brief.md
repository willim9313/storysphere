## 1. 目的與範圍

重新設計 `/books/:id/graph` 整頁。目標不是微調現有版面，而是允許重新安排資訊架構與互動模型，條件是：

1. **既有能力一項不能少**（見 §2 盤點）——可以改呈現方式、改入口位置，不能砍功能。
2. **加入 §5 的新功能**，其中「雙實體關係演變」是本輪重點。
3. 遵守 §6 的技術與視覺約束。

### 2.1 Lens × 群集模式的現況交互（實碼確認，重設計需明確定義）

現況三個 lens 與三種模式是「沒有設計過的正交」：

| | 個別 | 類型 | 社群 |
|---|---|---|---|
| 時間軸 | ✅ 完整生效 | ✅ 生效（聚合計數跟著變） | ⚠️ 半生效：成員清單跟章節走，但**派系劃分永遠用全書關係算**（後端支援 `?chapter=`，前端沒傳） |
| 認知視角 | ✅ 生效 | ❌ 無效果（樣式對準個別節點 id） | ❌ 完全沒接（FactionCanvas 未接收 epistemic 樣式） |
| 書籤 | ✅ 完整 | ⚠️ 點擊靜默無反應 | ⚠️ 同左（對比：社群 drill-in 點成員會自動切回個別模式，書籤點擊不會） |

隱性耦合：認知視角的「依賴章節」取自時間軸位置，時間軸在「全部」時靜默改用第 1 章。

**設計需回答**：lens 是「全域狀態、三模式都必須尊重」，還是「個別模式專屬工具」？現況是意外的中間態。概念分層可參考：時間軸=資料層 lens（改變抓回的圖）、認知視角=呈現層 lens、書籤=跨模式 id 清單。

## 3. 實測發現的 UX 痛點（重設計必須解決）

依嚴重度排序：

1. **高 degree 節點選取後標籤爆炸**：選「寇仲」時全部鄰居節點＋事件長標題同時顯示，大面積重疊不可讀。需要聚焦模式：非鄰居強力淡出、標籤依重要度/縮放層級限量。
2. **推論功能語意不明**（使用者實際回饋，見 §4）。
3. **孤兒節點以方陣漂浮**在主圖旁，看起來像 render bug。需收進「未連結實體」抽屜或預設隱藏。
4. **初始視野未 fit**：主圖偏左，右半空白。
5. **低縮放層級標籤全開**，糊成一片；需 zoom 門檻與重要度排序。
6. **認知視角效果太微弱**：只有虛線框差異，且無「已知 X/Y 實體」摘要；選單還會被視窗底部截斷。
7. **社群名稱是 generic 的 "Faction 1~10"**，全中文介面裡夾英文流水號，無語意（後端 `faction_service.py` 寫死 `f"Faction {i+1}"`，改名需動後端）。另注意社群檢視的資料範圍：faction 偵測**只對角色**做（正向關係 modularity clustering），非角色實體與未歸屬角色在此檢視中完全消失（實測 264 節點只剩約 31 個角色被分入派系）——這個「大量實體憑空不見」的狀態需要設計交代。
8. **個別檢視的邊無語意**：社群檢視有敵對/合作+權重，個別檢視全是同色灰線。
9. **節點大小（登場頻率）無圖例說明**。
10. 既知 bug：EntityDetailPanel 深度分析 header 巢狀 `<button>`（React hydration error）。

## 4. 重點重設計項目：推論功能

現況行為（實碼確認）：

- 「執行推論」= `POST /books/{id}/inferred-relations/run`，**作用層級是整本書**。演算法是純圖論（NetworkX Common Neighbors + Adamic-Adar），**不是 LLM**：找出「無直接關係但共同鄰居多」的實體對，取 top-N 當候選，AA 分數正規化為 confidence，關係型別依既有關係與實體型別 heuristic 建議。執行快、無 token 成本。
- 產出是 **pending 清單**，要在 InferredEdgePanel 逐條「採納 / 拒絕」；採納後升級為正式關係進入圖譜。
- 跑完後**同一顆按鈕變身**為「推論 N」顯示/隱藏 toggle——執行動作與顯示開關共用一個控制項。
- 另有「安全重跑」（只評新實體對）與「強制重跑」（重置所有已決定的紀錄，具破壞性）藏在面板內。

使用者實際困惑：**看不出按下去作用在什麼層級、會發生什麼、結果去了哪**。

設計要求：

- 執行（動詞、產生新資料）與顯示（狀態開關）必須是**兩個分離的控制項**。
- 執行前要能預知：作用範圍（全書）、會產生「待審核的推測關係候選」。
- 有 pending 項目時要有明確的待辦訊號（badge/計數），審核流入口清楚。
- 推測邊在圖上要與正式邊視覺區隔（現況是 warn 色，可沿用概念）。
- 強制重跑的破壞性（清掉所有已採納/拒絕決定）要在 UI 上被誠實表達，不只靠 confirm dialog。

## 5. 新功能

### F1（重點）雙實體關係演變動畫

選定兩個實體 A、B，以章節為步進單位，動畫呈現兩者關係的演變：直接關係邊的出現/變化、共同鄰居（尤其事件節點）的逐步累積。

**建議互動骨架**（Claude Design 可重構）：

- 進入方式：從 EntityComparePanel 升級，或圖上多選兩節點 → 「觀看關係演變」。
- 畫面：只顯示 A、B、與兩者的共同鄰居構成的子圖；底部章節步進器（◀ ▶ / 播放）；側欄顯示本章新增的連結與事件摘要。
- 每步進一章：新出現的邊/共同鄰居以動畫進場，既有元素保持原位。

**已知風險與對策**（設計時需納入）：

| 風險 | 說明 | 對策 |
|------|------|------|
| 共同鄰居爆量 | 主角對（寇仲×徐子陵）共同鄰居可達數十個事件，動畫不可讀 | 依共現權重排序、上限 ~15 個，其餘收進「+N」聚合節點 |
| 空演變 | 邊緣實體對可能整本書只有一次連結，動畫空洞 | 進入前先算變化量，不足時降級為靜態視圖＋提示 |
| 佈局跳動 | 每章重跑 layout 節點會亂跳 | 先以「全章節聯集子圖」算一次固定佈局，之後只做淡入/淡出 |
| 資料粒度 | 演變單位是章節 snapshot（`mode=chapter&position`），章內順序不可得 | 明確以「章」為步進單位，不承諾更細粒度。若該書已啟用 story mode（敘事順序時間軸），步進單位可切換為敘事順序——設計需預留這個切換，但 P1 以 chapter mode 為準 |
| 語意 vs 結構 | 圖只呈現結構變化，說不出「從陌生到結拜」這種質變 | 側欄章節摘要可先用既有事件標題堆疊；LLM 生成演變敘事列為 P2（成本/延遲/快取問題同深度分析） |
| 狀態互斥 | 與時間軸過濾、認知視角、群集模式的組合會爆炸 | 演變模式設計為**獨佔模式**（進入時暫停其他 lens），退出還原 |
| 請求數 | 逐章 snapshot = N 次請求 | 7 章可接受；長書需後端一次回傳 pair-evolution 子圖（新 endpoint，列為實作期決策） |

### F2 兩實體路徑追溯

A 與 B 無直接關係時，顯示連接兩者的最短關係鏈（經過哪些事件/中介者）。與 F1 同一個「實體對」模式的靜態子功能。

### F3 逐章成長播放

現有時間軸 slider 加自動步進播放，看整張圖譜逐章長大。與 F1 共用步進器設計語言。

### F4 深連結與匯出

- 選取實體、章節位置、群集模式寫入 URL query，分享/重整可還原。
- 匯出目前視圖為 PNG。

## 6. 技術與視覺約束

- **圖渲染是 Cytoscape**：canvas 繪製，設計上圖區內的節點/邊樣式自由度受限（無 DOM、無 CSS 動畫；顏色只吃 hex/rgb，不吃 oklch——`--graph-*` tokens 因此都是 hex）。
- **主題 token**：一律 `var(--*)`（`frontend/src/styles/tokens.css`），支援多主題（含 ink 深色），不可硬編碼色碼。
- **全中文 UI**（zh-TW），文案走 i18n（`frontend/src/i18n/locales/zh-TW/graph.json`）。
- **既有後端 API 優先**：§2 的能力都有現成 endpoint；F1 逐章 snapshot 可用現有 `GET /books/{id}/graph?mode=chapter&position=n`。新 endpoint 是實作期決策，設計不必遷就。
- **必守的設計 contracts**（源自現行規格，版面可全部重來，這幾條不行）：
  - 實體型別控制（篩選、圖例）一律涵蓋**完整 7 類**（角色/地點/組織/物品/概念/事件/其他），不得用子集。
  - 節點/邊顏色使用 `--graph-{type}-fill / -stroke / -label` token 體系（值為 hex）。
  - 群集模式的使用者偏好、時間軸位置、認知視角、書籤皆為 per-book 記憶（localStorage，key 不可改）。
  - 實作落地時由我們同步改寫 `docs/UI_SPEC.md` §3.6，設計端不需處理。
- **localStorage keys 必須保留**（UI_SPEC 明定，換版面不換 key）：`graph:${bookId}:timeline:*`、`graph:${bookId}:epistemic:*`、`graph:${bookId}:bookmarks`、`graph:${bookId}:clusterMode`。書籤是 per-browser 資料，F4 深連結不會帶著書籤走。
- **URL 深連結現況只有一半**：`?entity=` 已支援（自動選取實體），章節位置、群集模式、lens 狀態皆未入 URL——F4 是把既有機制補完，不是從零開始。
- 版面基準：1600×1000 桌面視窗；左側全域 sidebar（48px）與上方書籍導覽列為既有框架，不在本次重設計範圍。

## 7. 需要設計的畫面/狀態清單

1. 預設載入（個別檢視、全部章節、無選取）——含孤兒實體的處理
2. 節點選取（高 degree 角色）＋聚焦模式＋詳情面板
3. 類型檢視、社群檢視（含有語意的社群命名呈現；社群檢視需交代未歸屬角色與非角色實體的去向）
4. 推論：未執行 / 執行中 / 有 pending 待審 / 審核面板 / 顯示推測邊
5. 認知視角啟用中（含已知比例摘要）
6. 實體對模式：F1 演變動畫（含步進器、側欄）、F2 路徑追溯、空狀態
7. 時間軸章節過濾 + F3 播放中
8. 空圖/分析未完成的 onboarding 狀態（現有 GraphOnboardingHero 可重設計）

## 8. 開放問題（設計前不需定案，canvas 可提案）

- 工具列現況一列塞滿（搜尋+7 chips+3 模式+重置+推論+動畫模式），是否拆層級？
- LensCard（時間軸/視角/書籤）與右側詳情面板的地盤劃分是否維持左下/右側？右側面板現況四種內容（比較/推論審核/群集概觀/實體詳情）互斥搶位，重設計可重新分配。
- 「淡入/逐個」進場動畫是否值得保留為使用者可見選項？
- ~~社群命名的資料來源由後端決定~~ **已定案（2026-07-18，已另以補充訊息告知 Claude Design）**：派系名稱採錨點推導（以最核心成員命名，如「寇仲陣營」），不用 LLM、不做手動命名流程；名稱隨重算自動更新。
- 社群檢視曾有一份舊 handoff 規格（BACKLOG F-16）；本輪**不需參考、可自由重來**，與舊規格的對帳由實作端處理。
- 認知視角是否要把 misbeliefs（角色誤信清單，API 已回傳、圖譜頁未用）帶進視覺？
- ~~群集設定是否升級為可見控制？~~ **已定案（2026-07-18，已另以補充訊息告知 Claude Design）**：分群參數不做為使用者可見控制，預設隱藏（最多進階抽屜）；不需要「定型/儲存分群」流程，社群檢視永遠呈現當前 KG 的即時分群。

## 9. Canvas 對帳與實作裁決（2026-07-18，與使用者逐項確認）

對照回收的 canvas（`知識圖譜重設計.dc.html`）發現的偏差與逐項裁決。**實作時本節優先於 canvas**：

1. **標籤與密度策略（實作補強）**：canvas 以 36 節點/44 邊示意，未演示真實密度（264 節點/506 關係，其中事件 62 個且標題為長句）。實作必須補：標籤 top-N 上限（依 degree/重要度排序，聚焦模式只顯示焦點＋鄰居中的前 N）、事件長標題單行截斷（ellipsis）；zoom 門檻沿用 canvas 邏輯。設計稿無法呈現的密度體感，以真實資料在實作階段調校。
2. **F1 側欄章節摘要降級**：canvas 內的逐章敘事文案視為 placeholder。P1 實作為「本章新增」事件標題堆疊；LLM 敘事維持 P2 不提前。
3. **空狀態補齊（canvas 未涵蓋，依 §7-6 實作）**：
   - F1 進入前檢查變化量（共同鄰居過少或跨章無變化）→ 降級為靜態共同鄰居視圖＋提示「兩實體關係在本書中無明顯演變」。
   - F2 無路徑可連 → 空狀態說明＋建議（例如開啟推測邊後重試）。
4. **搜尋下拉保留**：canvas 只畫了輸入框，不代表刪除功能。保留現有 SearchDropdown 分組下拉（實體/章節/段落內文），視覺套用新設計卡片樣式。
5. **認知視角 fallback 定案**：時間軸「全部」→ 以**全書終局**計算認知（採 canvas 行為，變更現行「第 1 章」邏輯），說明文字照 canvas。
6. **敘事順序 toggle 需 gated**：`detect-timeline` 判定 story mode 不可行時顯示鎖定態（disabled＋tooltip），canvas 未畫，實作補。
7. **忽略 canvas 頂欄的 Warm/Ink 主題切換鈕**：canvas 預覽用控制、非產品功能；主題切換唯一入口仍為設定頁（ThemeContext 規範）。
8. **預設裁決（可推翻）**：事件節點點選沿用既有 EventDetailPanel 的內容欄位，套用新右側面板樣式；canvas 未區分事件與一般實體詳情。
9. canvas 內 mock 數字（如強制重跑「12 筆」）一律以實際資料為準。
